# Доклад для защиты курсовой работы

## Тема
Информационная система мотивации накопления средств для кредитной цели с учетом уровня дохода пользователя.

## Цель проекта
Разработать веб-приложение, которое:
- принимает финансовые данные пользователя;
- вычисляет рекомендуемую сумму ежемесячных накоплений;
- учитывает уровень дохода;
- сохраняет историю расчетов в базе данных;
- предоставляет авторизацию и экспорт данных.

## Кратко про архитектуру
- Frontend: React + FSD (`app`, `pages`, `widgets`, `entities`, `shared`).
- Backend: Node.js + Express (REST API).
- DB: SQLite.
- Авторизация: регистрация/вход по email+пароль, токен в заголовке `Authorization`.

## Отдельный блок для заучивания (чтобы защитить)

### 1) Речь на 1-2 минуты
«Я реализовал веб-приложение для мотивации накоплений на кредит.  
Пользователь регистрируется, вводит доход, расходы и срок.  
Сервер определяет уровень дохода и выбирает случайный процент накопления в допустимом диапазоне.  
Далее рассчитывается безопасная сумма ежемесячных накоплений с учетом расходов.  
Результат сохраняется в SQLite и отображается в истории.  
Архитектура фронтенда построена по FSD: `app`, `pages`, `widgets`, `entities`, `shared`.  
Есть экспорт истории в JSON и CSV.  
Таким образом, система решает задачу персональной финансовой мотивации и хранения аналитики по пользователю.»

### 2) 10 вопросов комиссии и короткие ответы
1. **Почему FSD?**  
Потому что FSD разделяет ответственность по уровням и упрощает масштабирование интерфейса.

2. **Почему SQLite?**  
Легкая файловая БД, не требует отдельного сервера, подходит для учебного проекта.

3. **Как учитывается доход?**  
Через уровни `low/middle/high`, у каждого свой диапазон процента накоплений.

4. **Зачем случайность?**  
Для мотивационного эффекта и вариативности рекомендаций в пределах безопасных границ.

5. **Как защищены пароли?**  
Пароли хешируются через `scrypt` и соль, в БД не хранится открытый пароль.

6. **Как работает авторизация?**  
После входа выдается токен, фронтенд передает его в `Authorization: Bearer ...`.

7. **Как разделяются данные пользователей?**  
Все расчеты имеют `user_id`; выборка идет только по текущему пользователю.

8. **Что делает middleware?**  
Проверяет токен, извлекает пользователя и кладет его в `req.user`.

9. **Какие есть экспорты?**  
JSON и CSV через отдельные защищенные endpoint'ы.

10. **Как проверить работоспособность?**  
Запустить backend и frontend, зарегистрироваться, сделать расчет, скачать экспорт.

### 3) Технические акценты для уверенной защиты
- `recommendationService` содержит чистую бизнес-логику расчета.  
- Репозитории изолируют SQL от роутов.  
- `authMiddleware` централизует проверку доступа.  
- В `MainPage` собран пользовательский поток: вход -> расчет -> история -> экспорт.  
- В проекте есть валидации входных данных и обработка ошибок.

### 4) Что можно сказать как «планы развития»
- заменить in-memory токены на JWT с временем жизни;
- добавить refresh-token и logout на сервере;
- добавить диаграммы накоплений;
- добавить экспорт в PDF/XLSX.
